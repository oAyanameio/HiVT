# 实验问题阶段性解决方案文档

## 1. 文档目的

本文档只保留当前仍会影响实验推进的事项：未解决问题、当前配置、下一步实验顺序与执行标准。

本阶段的核心原则是：

```text
先确认 risk score 是否真的有增量价值，再决定是否扩大训练和论文包装。
```

## 2. 当前实验状态

当前插件已经完成了结构和训练链路的最小闭环：

- `mode_risk` / `scene_risk` / `reranked_pi` 已接入 HiVT。
- FDE / ADE / miss / conflict / off-road 自动标签已接入。
- `L_pred + L_mode + L_scene` 联合训练已跑通。
- `AUROC` / `BrierScore` / `ECE` 等可靠性指标已接入验证流程。
- 小预算 baseline/plugin 对照实验已经完成。
- 宿主原始 `pi` 的 naive baseline、freeze-backbone 对照、reranking-only 对照已经补齐。

阶段性实验结果表明：

- reliability head 已表现出可判别的失败风险估计能力。
- 当前联合训练设置会拉低宿主原始预测指标。
- freeze-backbone 可以基本保住宿主预测性能。
- 当前 `mode_risk` 已有较强 FDE 排序能力，但现有 reranking 公式还不能带来收益。

## 3. 当前未解决问题

### 3.1 宿主预测指标退化

当前最直接的问题是联合训练会损伤宿主预测指标：

```text
baseline: val_minADE=0.5738, val_minFDE=0.8031, val_minMR=0.0508
plugin:   val_minADE=0.6110, val_minFDE=0.8664, val_minMR=0.0703
```

其中 `MR` 的退化最值得优先诊断。

### 3.2 当前 reranking 映射无效

之前缺失的宿主原始 `pi` baseline 已经补齐，结果是：

```text
naive_mode_AUROC = 0.7000
naive_mode_AUPRC = 0.7140
```

这说明 reliability 分支相对原始 `pi` 有明显增量。

但当前真正未解决的问题变成了：

```text
reranked score = pi - alpha * risk
```

在 `alpha=0.2/0.5/1.0` 下都使 `minADE/minFDE/MR` 小幅变差。

### 3.3 Scene 预测存在正偏差

当前记录中：

```text
val_scene_risk_pred_mean   = 0.6350
val_scene_risk_target_rate = 0.3945
```

补充诊断后还看到：

```text
scene_BrierScore = 0.2541
scene_ECE        = 0.2356
```

这说明 scene head 还需要单独做校准诊断，不能只看 mode ECE。

### 3.4 Off-road 仍然只是弱代理

当前 off-road 仍基于 lane centerline distance，而不是 drivable-area polygon / raster，因此暂时只适合作为辅助日志与弱安全代理，不应重新并入主监督。

## 4. 当前阶段目标

当前阶段的目标只有三项：

- 确认 reliability 分支相对宿主原始 `pi` 是否真的有增量价值。
- 确认风险分数是否具备可用于 reranking 的排序能力。
- 确认如何在不继续拉低宿主 `minADE / minFDE / MR` 的前提下使用 reliability 分支。

其中前两项已经得到正向答案：

- 相对宿主原始 `pi`，reliability 分支在 failure detection 上有真实增量；
- `mode_risk` 与 `FDE` 的 Spearman 相关性已经达到 `0.7482`。

当前最核心的问题收敛为：

- 如何设计更合理的 reranking 映射；
- 是否需要对 scene head 再做校准而不是继续堆训练。

## 5. 当前执行路线

### 5.1 冻结主干 / Reranking-only 验证版（已完成）

当前最优先的是避免 reliability loss 干扰已训练好的预测主干，验证风险分数本身是否有部署价值：

- 冻结 HiVT backbone，只训练 `reliability_module`。
- 保持同一 checkpoint，比较 `original pi` 与 `reranked_pi`。
- 暂不提高 `rank_loss_weight` / `calib_loss_weight`。

当前结果是：

- freeze-backbone 已经验证能保住宿主 baseline 的 `minADE / minFDE / MR`；
- reranking 确实改变了 top-1 mode；
- 但当前 `pi - alpha * risk` 公式没有改善 mode selection，反而带来小幅退化。

### 5.2 多任务论文版

在 freeze backbone / reranking-only 验证通过后，再把风险源拆成多个任务：

- prediction failure risk
- conflict risk
- off-road risk
- scene risk

目标是增强可解释性和论文完整度。但这一版实现和调参更复杂，应放在 freeze backbone / reranking-only 验证之后推进。

## 6. 已采用标签配置

### 6.1 Mode Risk 主监督

当前已经不再使用五类标签并集作为默认主监督。

已采用的主 `mode_targets` 是预测失败标签：

```text
y_mode = 1(FDE_k > tau_fail)
```

保留但暂不作为默认的备选策略：

```text
y_mode = 1(FDE_k > tau_fail) OR 1(MR_k = 1)
```

其中 `MR_k` 应对应明确的 miss threshold。如果 `tau_fail` 与 `miss_threshold` 都基于 FDE，需要避免两个阈值在 OR 之后退化成单一较小阈值。当前实现中：

- `miss_targets`: 与 MR 指标一致，表示是否超过 miss threshold。
- `fde_targets`: 更宽或更严格的预测失败阈值。
- `mode_targets`: 本阶段优先选一个清晰定义，而不是盲目 OR 多个阈值。
- `risk_fde_threshold` 与 `risk_miss_threshold` 必须分工明确，不能只是在代码里重复定义两次终点误差阈值。

当前采用的阈值关系：

```text
risk_fde_threshold = 2.0
risk_miss_threshold = 4.0
```

含义建议固定为：

- `risk_fde_threshold`: 用于定义“预测失败但不一定彻底 miss”的 mode-level failure。
- `risk_miss_threshold`: 用于定义“明显 miss GT”的更强失败事件。

如果 `risk_miss_threshold > risk_fde_threshold`，则：

```text
fde_or_miss = fde_only
```

在逻辑上几乎退化。因此当前默认：

- 主监督默认直接使用 `fde_only`。
- `miss_targets` 先保留为辅助统计和日志。
- `fde_or_miss` 只作为显式对照策略保留。

短期内，`ADE` 不进入主风险标签，只作为辅助日志保留。原因是 ADE 与 FDE 混合后会让失败定义变得不够清晰。

### 6.2 Conflict 标签

Conflict 已经从宽松的轨迹最小距离代理，改成更严格的时序碰撞代理。

当前定义：

```text
conflict = same_time_distance(target_mode_k(t), neighbor_future_gt(t)) < tau_col
           AND close_frames >= m
           AND target_future_valid(t)
           AND neighbor_future_valid(t)
```

当前参数：

```text
tau_col = 1.0m
m = 2 frames
```

当前采用：

- `target actor` 的第 `k` 条预测 future
- 对比同 scene 下每个有效邻车的 `GT future`
- 仅做同一时间步距离比较
- 对每个邻车只在双方 future mask 同时有效的步上计算
- 若任一邻车满足持续接近条件，则该 `target mode k` 记为 conflict

当前比较范围：

- 只比较 target actor 的预测轨迹与有效邻车 future。
- 暂不做全场景所有 actor 两两 mode 比较。
- 使用 actor valid mask、target future mask、neighbor future mask。
- 可过滤距离过远或动态状态无效的邻车。

不建议第一阶段使用下面这种定义作为主实现：

```text
target prediction vs neighbor prediction
all modes vs all modes
all actors vs all actors
```

这些定义更适合作为后续诊断或辅助统计，因为它们会显著抬高密集交通场景里的 `conflict_risk_target_rate`。

短期内，`conflict` 不进入主 `mode_targets`，只作为：

- `conflict_risk_target_rate`
- 安全代理指标
- 后续辅助 head 标签

这样可以避免 conflict 再次污染主风险监督。

### 6.3 Off-road 标签

当前 off-road 标签不建议作为主监督。

短期策略：

- 从主 `mode_targets` 中移除。
- 保留 `offroad_risk_target_rate` 日志。
- 在文档或代码注释中明确它是 lane support proxy，而不是真正 drivable-area violation。

中期优化：

- 使用 drivable-area polygon 或 raster 判断。
- 引入 lane segment 宽度，而不是只看 centerline distance。
- 要求连续若干 future step 越界，避免单点噪声触发。

### 6.4 Scene Risk 聚合

当前已经不再默认使用全场景 hard max：

```text
y_scene = max over all nodes/modes y_mode
```

当前采用 target-centric 的比例聚合，避免“target actor 只要任意 mode 为正，整个 scene 就为正”这种过密定义。

已验证的策略如下：

```text
历史候选 A:
y_scene = 1(target actor best/min-FDE mode failed)
```

```text
当前采用 B:
y_scene = 1(target actor positive mode rate > rho)
```

不推荐第一阶段默认使用：

```text
y_scene = 1(target actor has any positive mode)
```

因为在多模态设置下，只要目标车有一部分 mode 明显偏离 GT，这个定义就很容易再次把 `scene_risk_target_rate` 推高到接近 `1.0`。

更具体地说，推荐 scene policy 语义改成：

```text
target_best_mode_fail
```

或：

```text
target_mode_rate
```

如果需要保留更强的 scene-level 概念，可以后续使用比例聚合：

```text
y_scene = 1(high_risk_mode_rate > rho)
```

当前采用 `target_mode_rate`，`rho=0.67`。它在宿主 checkpoint 审计中得到 `scene_positive_rate=0.4063`，比 `target_best_mode_fail` 更适合当前训练闭环。

### 6.5 Target Actor 索引与标签归属

本轮修复中，所有 target-centric 标签策略都不能只停留在文档层面，必须在实现上显式传入目标 actor 索引。

建议在 `build_reliability_targets` 及相关子函数中新增：

```text
agent_index
```

并约定：

- `agent_index` 来自每个 graph 的目标车辆索引。
- `target_to_neighbors` conflict 策略依赖 `agent_index` 才能正确找到 target actor。
- `target_best_mode_fail` / `target_mode_rate` 这类 scene policy 也依赖 `agent_index`。
- 若缺少 `agent_index`，则 target-centric 策略应直接报错或显式回退，不应静默退化成全 actor 聚合。

## 8. 当前训练策略

### 8.1 当前配置

当前推荐配置已经固定为：

```text
mode_target_policy = fde_only
scene_target_policy = target_mode_rate
risk_scene_rate_threshold = 0.67
risk_fde_threshold = 2.0
risk_miss_threshold = 4.0
risk_conflict_threshold = 1.0
risk_conflict_min_frames = 2
risk_conflict_scope = target_to_neighbors
reliability_loss_weight = 1.0
scene_loss_weight = 0.2
rank_loss_weight = 0
calib_loss_weight = 0
rerank_alpha = 0
```

该配置已经通过 `64/16` 与 `128/32` 小预算实验验证，标签分布不再塌缩，risk head 已具备可判别性。

### 8.2 后续可打开项

后续不应直接把所有复杂项一起打开。推荐顺序是：

```text
1. 无训练诊断：naive baseline / risk-FDE correlation / scene calibration
2. freeze backbone，只训练 reliability_module
3. reranking-only，对比 original pi 与 reranked_pi
4. 小权重打开 rank_loss_weight = 0.05 ~ 0.1
5. calib_loss_weight 继续从 0.0 起步
6. rerank_alpha 从 0.2 逐步试到 0.5
```

`scene_loss_weight` 仍不宜过大，因为 scene label 比 mode label 更粗，权重过大可能干扰 mode-level 学习。

另外需要明确当前实现边界：

- 当前只有 `mode_risk` 与 `scene_risk` 两个训练头。
- `conflict_targets` 目前只作为辅助标签与日志统计，不参与单独 BCE 训练。
- 因此当前 “interaction-aware” 的实现支点主要是 `InteractionContextEncoder`，而不是一个独立的 conflict head。

### 8.3 当前指标解释顺序

当前指标解释顺序应为：

1. 插件是否不破坏宿主 `minADE / minFDE / MR`。
2. `original pi` 与 `reranked_pi` 的 mode selection 是否有差异。
3. `AUROC / AUPRC / BrierScore / ECE` 是否维持可解释。
4. risk score 是否对高 FDE / miss mode 有更高响应。
5. shift 条件下 risk score 是否随 failure rate 上升。

## 9. 实验闭环设计

### 9.1 当前已有实验依据

当前已有：

- 标签体检：基于宿主 `HiVT-64` checkpoint 完成。
- `64/16` 小预算训练验证：标签分布和 risk head 均恢复可解释。
- `128/32` 小预算对照：可靠性指标稳定，但联合训练轻微损伤宿主预测指标。

### 9.2 无训练诊断结果（已完成）

本轮已经补齐三项低成本诊断：

1. `naive baseline AUROC / AUPRC`

   直接使用宿主原始 mode score 构造风险代理：

   ```text
   risk_proxy_naive = -log softmax(pi_k)
   ```

   并对 `FDE > 2.0m` 标签计算 AUROC / AUPRC。

   结果：

   ```text
   naive_mode_AUROC = 0.700041
   naive_mode_AUPRC = 0.714026
   ```

   结论：

   - 宿主原始 `pi` 确实自带一定 failure detection 能力；
   - 但与 reliability 分支已有的 `0.8735 / 0.8358` 相比仍有明显差距。

2. `risk score vs FDE` 排序相关性

   在同一验证集上计算：

   ```text
   Spearman(mode_risk, FDE)
   ```

   结果：

   ```text
   Spearman(mode_risk, FDE) = 0.748202
   ```

   结论：

   - 当前 `mode_risk` 已具备较强 mode 排序价值；
   - 因此 reranking 无收益时，优先应怀疑重排公式，而不是先怀疑 risk head 完全没学到东西。

3. `scene` 校准诊断

   当前记录显示：

   ```text
   val_scene_risk_pred_mean = 0.6350
   val_scene_risk_target_rate = 0.3945
   ```

   补充结果：

   ```text
   scene_BrierScore = 0.254131
   scene_ECE        = 0.235648
   ```

   结论：

   - scene head 目前仍有明显校准偏差；
   - 当前不适合继续提高 scene loss 权重，而应先做更稳的 scene 映射与校准分析。

### 9.3 Reranking 单独验证结果（已完成）

使用同一个 checkpoint 分别评估：

```text
original pi
reranked_pi
```

本轮结果如下：

| alpha | reranked minADE | reranked minFDE | reranked MR | top-1 change rate |
|---:|---:|---:|---:|---:|
| 0.2 | 1.201279 | 2.585675 | 0.437500 | 0.023438 |
| 0.5 | 1.209395 | 2.598438 | 0.445312 | 0.074219 |
| 1.0 | 1.219001 | 2.613258 | 0.449219 | 0.128906 |

相对于同批数据上的 `original pi`：

```text
original minADE = 1.198633
original minFDE = 2.575924
original MR     = 0.425781
```

当前结论是：

- reranking 确实会改动 top-1 mode；
- 但现有 `pi - alpha * risk` 公式在 `alpha=0.2/0.5/1.0` 下都没有收益；
- 因此下一步不应继续重复同类训练，而应改公式。

### 9.4 Freeze Backbone 验证结果（已完成）

冻结 HiVT backbone，只训练 `reliability_module` 后：

```text
val_minADE = 0.574
val_minFDE = 0.803
val_minMR  = 0.0508
```

与宿主 baseline 基本一致。

这说明：

- freeze-backbone 是当前更合理的训练入口；
- reliability 分支本身不必然破坏宿主预测；
- 联合训练的主要问题是 backbone 受额外损失干扰。

### 9.5 下一步：改进 reranking 公式

当前最值得继续做的不是扩大训练，而是做轻量但信息密度更高的 reranking 公式对照：

1. 不再使用 `pi - alpha * risk` 作为主公式，因为它把未归一化 logit 和概率直接线性相减，量纲不一致；
2. 优先对照概率空间公式：

   ```text
   score_k = log_softmax(pi)_k + log(1 - mode_risk_k + eps)
   ```

   或等价地：

   ```text
   P(select k) ∝ softmax(pi)_k * (1 - mode_risk_k)
   ```

3. 再额外对照 `log_softmax(pi) - alpha * normalized_risk` 这类近似版本，作为弱基线；
4. 只在 top-k 候选 mode 内重排，避免把明显弱 mode 抬升；
5. 记录每种公式下的 `top1 change rate / minFDE / MR / minADE`。

这里需要明确：当前 reranking 的主要问题不是 `alpha` 没调好，而是原始公式本身缺乏统一的概率语义。

### 9.6 下一步：Conflict 标签重新审计

当前两次保留 run 的：

```text
conflict_risk_target_rate ≈ 0.0007
```

这个量级已经不能简单解释为“合理稀疏”，而更接近“标签几乎未触发”。因此下一步需要优先做阈值审计，而不是继续把 `interaction-aware conflict` 当成已验证结论。

建议按下面顺序做：

1. 固定当前 `target_prediction vs neighbor_GT_future` 定义；
2. 扫描 `risk_conflict_threshold = 1.0 / 1.5 / 2.0 / 3.0`；
3. 固定 `risk_conflict_min_frames = 2`；
4. 记录 `conflict_risk_target_rate` 是否能进入 `5%~15%` 的可训练区间；
5. 若放宽阈值后语义过于宽松，则在论文中明确将其降级为“安全代理日志”，而不是主贡献。

因此当前论文表述边界也要同步收紧：

- `InteractionContextEncoder` 仍然可以保留为交互上下文编码器；
- 但 `interaction-aware conflict reliability` 目前不能写成已被 conflict 监督直接支撑的主贡献。

### 9.7 下一步：Scene 校准单独处理

scene_risk_pred_mean = 0.6350

这说明 scene head 当前存在明显过置信和校准失配。现阶段更合理的判断是：

- 但当前概率输出不适合直接作为下游决策概率使用。

因此下一步应紧接在 reranking 公式之后：

1. 补 scene reliability curve 或 bin-level 统计；

4. 在 scene 校准修复前，不把 `scene_risk` 概率直接用于部署决策论证。

## 11. 推荐实施顺序

### 11.1 第一轮：改进 reranking 公式（概率空间）

1. 保持 freeze-backbone checkpoint 不变。
2. 新增概率空间公式：

   ```text
   log_softmax(pi) + log(1 - risk + eps)
   ```

3. 再对照 `log_softmax(pi) - alpha * normalized_risk`。
4. 扫描 `alpha` 与 `top-k` 范围。
5. 先找到“不劣化”的重排版本，再讨论收益。

### 11.2 第二轮：Conflict 标签 audit

1. 固定现有 conflict 定义。
2. 扫 `risk_conflict_threshold`。
3. 判断 conflict 标签能否进入可训练区间。
4. 若不能，则正式降级为日志 proxy，不再作为主贡献论据。

### 11.3 第三轮：Scene head 校准分析

1. 检查 scene head 输入聚合是否过粗。
2. 补 scene reliability curve 或分桶统计。
3. 视情况加入 temperature scaling 或其他后处理。
4. 在校准修复前，不把 scene 概率写成可直接使用的决策量。

### 11.4 第四轮：联合训练 MR 退化来源分析

1. 比较联合训练与 freeze-backbone 的 `L_pred` 变化。
2. 补 backbone 关键模块梯度范数或等价诊断。
3. 区分退化主要来自 mode selection 破坏还是轨迹回归质量下降。
4. 这一步完成前，freeze-backbone 只能算规避方案，不能算机制解释。

### 11.5 第五轮：中等预算与 Shift Evaluation

1. 在前四轮结论稳定后，再跑中等预算完整对照。
2. 至少确认 `plugin > naive baseline` 的相对排序在更大数据量下稳定。
3. 再加入 shift 扰动，比较 failure rate 上升时 `pi` 与 `mode_risk` 的响应差异。

### 11.6 第六轮：恢复论文级实验

1. 加入 shift evaluation。
2. 重新跑 baseline/plugin 对照。
3. 做必要消融，至少包括去掉 `InteractionContextEncoder` 的对照。
4. 最后再考虑多 backbone 泛化。

## 12. 论文表述调整

当前阶段不应表述为：

```text
插件显著提升 HiVT 预测精度
```

更稳妥的表述是：

```text
在基本不破坏宿主预测性能的前提下，引入可判别的 mode-level 与 scene-level 失败风险估计，并通过 risk-aware reranking 和 shift-aware evaluation 体现部署价值。
```

论文创新点应聚焦于：

- 自动风险标签协议。
- mode-level + scene-level 双层可靠性。
- interaction-aware 上下文编码与 target-centric 风险聚合。
- shift-aware reliability evaluation。
- risk-aware reranking。
- plug-in 形式接入宿主轨迹预测模型。

这里需要注意表述边界：

- 当前实现已稳定支撑的是：标签协议、双层风险建模、插件式接入。
- `interaction-aware conflict proxy` 目前更适合作为辅助安全代理与日志项，而不是已被强监督验证的主贡献。
- `shift-aware evaluation` 与 `reranking` 还需要补实验，现阶段不应写成既成结论。
- 当前 scene head 存在明显校准失配，不能把其概率输出直接写成可部署决策量。

## 13. 当前不建议做的事情

在确认 freeze backbone / reranking-only 不破坏宿主预测指标之前，不建议：

- 直接跑长时间正式训练。
- 直接做 QCNet / MTR / Wayformer 泛化。
- 直接加复杂多 head 并提高损失权重。
- 宣称插件已经提升预测精度。

这些工作应在确认可靠性分支不会拉低宿主预测指标之后再推进。

## 14. 简短结论

当前主要问题已经收敛为：

- 联合训练下宿主预测指标出现明确退化，尤其 `MR: 0.0508 -> 0.0703`；
- 但 freeze-backbone 已证明这种退化可以避免；
- reliability 分支相对宿主原始 `pi` 已有明确增量；
- `mode_risk` 已具备较强 FDE 排序能力；
- 当前真正卡住的是 reranking 公式无效，以及 scene head 仍有明显校准偏差。

下一步最优先的工作不再是继续堆训练，而是基于 freeze-backbone checkpoint：

1. 先把 reranking 改到概率空间；
2. 再确认 conflict 标签到底能不能形成可训练监督；
3. 然后单独定位 scene 校准偏差；
4. 最后再补联合训练退化来源分析。

只有这几步走通后，再扩大到中等预算、shift evaluation 和更完整论文实验。
