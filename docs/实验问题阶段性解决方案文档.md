# 实验问题阶段性解决方案文档

## 1. 文档目的

本文档用于记录当前 `HiVT + reliability plugin` 实验中暴露出的阶段性问题，并给出一套可执行的修复方案。

当前重点不是继续扩大训练规模，也不是马上做多 backbone 泛化，而是先解决 reliability supervision 的塌缩问题。只有当风险标签恢复可判别性之后，后续的 risk-aware reranking、shift evaluation 和论文级实验才有解释意义。

本阶段的核心原则是：

```text
先让 risk label 可学，再让 risk score 有用，最后再让模块更完整、更像论文。
```

## 2. 当前实验状态

当前插件已经完成了结构和训练链路的最小闭环：

- `mode_risk` / `scene_risk` / `reranked_pi` 已接入 HiVT。
- FDE / ADE / miss / conflict / off-road 自动标签已接入。
- `L_pred + L_mode + L_scene` 联合训练已跑通。
- `AUROC` / `BrierScore` / `ECE` 等可靠性指标已接入验证流程。
- 小预算 baseline/plugin 对照实验已经完成。

阶段性实验结果表明：

- 插件没有明显破坏宿主 HiVT 的预测性能。
- 插件尚未证明能稳定提升预测精度。
- 更关键的是，当前风险标签几乎全为正样本，risk head 已经出现塌缩。

## 3. 核心问题诊断

### 3.1 Risk 标签塌缩

当前实验中出现了如下现象：

- `train_mode_risk_target_rate = 1.0`
- `val_mode_risk_target_rate = 1.0`
- `train_conflict_risk_target_rate = 1.0`
- `val_conflict_risk_target_rate = 1.0`
- `train_scene_risk_target_rate = 1.0`
- `val_scene_risk_target_rate = 1.0`
- `mode_risk_pred_mean` 和 `scene_risk_pred_mean` 接近 `1.0`

这说明当前风险监督没有正负样本区分度。模型最优策略变成直接预测所有 mode 和所有 scene 都是高风险。

在这种情况下，`AUROC` / `BrierScore` / `ECE` 不能被解释为插件具备可靠风险估计能力，因为标签本身已经失去判别性。

### 3.2 Conflict 标签过密

当前 conflict 标签容易把密集场景全部判为冲突，主要原因包括：

- 使用轨迹之间的最小距离作为代理，定义过宽。
- 缺少严格的同一时间步碰撞判断。
- 缺少持续帧数约束。
- 可能包含过多无关 actor 之间的两两比较。
- 没有充分利用有效 actor / 有效 future mask。

因此 `conflict_risk_target_rate` 直接变成 `1.0`，并进一步污染主 `mode_targets`。

### 3.3 Off-road 标签仍是弱代理

当前 off-road 标签主要依赖 lane centerline distance，而不是 drivable-area polygon 或 raster。

这个代理存在两个问题：

- 可能把合法变道、路口区域、lane-adjacent 轨迹误判为 off-road。
- 也可能漏掉真实驶出可行驶区域的轨迹。

因此当前 off-road 不适合作为主风险监督，只适合作为辅助日志或弱安全代理。

### 3.4 Scene Risk 聚合过激

当前 scene risk 采用类似 `max over node/mode risk` 的聚合方式。

在多 actor、多 mode 场景中，只要任意 actor 的任意 mode 被判为高风险，整个 scene 就会被标成高风险。这会导致 `scene_risk_target_rate` 很容易接近 `1.0`。

### 3.5 主风险标签被子标签污染

当前主风险标签近似为：

```text
mode_targets = max(fde, ade, miss, conflict, offroad)
```

这种并集式定义在早期阶段风险很高。只要其中一个子标签过密，主 `mode_targets` 就会塌缩成全正。

## 4. 阶段性优化目标

本阶段的目标不是直接证明插件提升 HiVT 精度，而是先恢复 risk supervision 的有效性。

第一阶段验收目标：

- `mode_risk_target_rate` 不再接近 `1.0` 或 `0.0`。
- `scene_risk_target_rate` 不再接近 `1.0` 或 `0.0`。
- `conflict_risk_target_rate` 明显低于当前的全正状态。
- `mode_risk_pred_mean` 不再逼近 `1.0`。
- `AUROC` / `AUPRC` / `BrierScore` / `ECE` 具备基本解释意义。

经验上可以先参考下面的分布范围：

- `mode_risk_target_rate`: 约 `0.15 ~ 0.45`
- `scene_risk_target_rate`: 约 `0.2 ~ 0.6`

这些范围不是硬性目标。标签必须先有物理意义，再检查分布是否适合训练。

## 5. 总体修复路线

推荐按三阶段推进。

### 5.1 保守修复版

先不大改网络结构，只修改标签定义和聚合策略。

目标：

- 快速验证塌缩是否主要来自标签定义。
- 恢复 mode-level 和 scene-level 风险监督的可判别性。
- 保持当前训练链路稳定。

这是当前应优先执行的路线。

### 5.2 排序增强版

在标签分布恢复正常后，再加入：

- `RiskRankLoss`
- `risk-aware reranking`
- reranked/original `pi` 对照评估

目标是验证风险分数是否能改善 mode selection，而不仅仅是输出一个风险概率。

### 5.3 多任务论文版

在主监督稳定后，再把风险源拆成多个任务：

- prediction failure risk
- conflict risk
- off-road risk
- scene risk

目标是增强可解释性和论文完整度。但这一版实现和调参更复杂，不适合在标签塌缩尚未解决时优先推进。

## 6. 具体标签修改方案

### 6.1 Mode Risk 主监督

当前不建议继续使用五类标签并集作为主监督。

第一版建议将主 `mode_targets` 改为预测失败标签：

```text
y_mode = 1(FDE_k > tau_fail)
```

或：

```text
y_mode = 1(FDE_k > tau_fail) OR 1(MR_k = 1)
```

其中 `MR_k` 应对应明确的 miss threshold，例如 `FDE_k > 2.0m`。如果 `tau_fail` 与 `miss_threshold` 都基于 FDE，需要避免两个阈值在 OR 之后退化成单一较小阈值。实现时应明确：

- `miss_targets`: 与 MR 指标一致，表示是否超过 miss threshold。
- `fde_targets`: 更宽或更严格的预测失败阈值。
- `mode_targets`: 本阶段优先选一个清晰定义，而不是盲目 OR 多个阈值。

建议先扫描：

```text
tau_fail = 2.0 / 3.0 / 4.0 / 5.0
```

选择标准：

- 标签语义明确，能代表预测失败。
- 正样本率不接近 `0` 或 `1`。
- 训练后 risk score 对失败 mode 有区分度。

短期内，`ADE` 不进入主风险标签，只作为辅助日志保留。原因是 ADE 与 FDE 混合后会让失败定义变得不够清晰。

### 6.2 Conflict 标签

Conflict 应从宽松的轨迹最小距离代理，改成更严格的时序碰撞代理。

推荐定义：

```text
conflict = same_time_distance(i, j, t) < tau_col
           AND close_frames >= m
           AND valid_actor_i
           AND valid_actor_j
```

推荐初始参数：

```text
tau_col = 1.0 ~ 1.5m
m = 2 ~ 3 frames
```

比较范围建议收窄：

- 第一版只比较 target actor 的预测轨迹与有效邻车 future。
- 暂不做全场景所有 actor 两两 mode 比较。
- 使用 actor valid mask 和 future valid mask。
- 可过滤距离过远或动态状态无效的邻车。

短期内，`conflict` 不进入主 `mode_targets`，只作为：

- `conflict_risk_target_rate`
- 安全代理指标
- 后续辅助 head 标签

这样可以避免 conflict 再次污染主风险监督。

### 6.3 Off-road 标签

当前 off-road 标签不建议作为主监督。

短期策略：

- 从主 `mode_targets` 中移除。
- 保留 `offroad_risk_target_rate` 日志。
- 在文档或代码注释中明确它是 lane support proxy，而不是真正 drivable-area violation。

中期优化：

- 使用 drivable-area polygon 或 raster 判断。
- 引入 lane segment 宽度，而不是只看 centerline distance。
- 要求连续若干 future step 越界，避免单点噪声触发。

### 6.4 Scene Risk 聚合

当前不建议使用全场景 hard max：

```text
y_scene = max over all nodes/modes y_mode
```

第一版建议改成 target-centric：

```text
y_scene = 1(target actor best/min-FDE mode failed)
```

或：

```text
y_scene = 1(mean target actor mode risk > rho)
```

如果需要保留更强的 scene-level 概念，可以后续使用比例聚合：

```text
y_scene = 1(high_risk_mode_rate > rho)
```

推荐第一阶段优先使用 target actor 定义，因为它与 HiVT 当前 `minADE` / `minFDE` / `MR` 的评估口径更一致，也更不容易被无关邻车污染。

## 7. 代码修改落点

### 7.1 `models/reliability.py`

优先修改以下函数：

- `compute_mode_risk_targets`
- `compute_miss_risk_targets`
- `compute_conflict_risk_targets`
- `compute_offroad_risk_targets`
- `compute_scene_risk_targets`
- `build_reliability_targets`
- `summarize_reliability_targets`

建议新增或调整的能力：

- 支持 `mode_target_policy`，例如 `fde_only`、`miss_only`、`fde_or_miss`、`all_union`。
- 支持 `scene_target_policy`，例如 `target_actor`、`target_mode_rate`、`scene_rate`、`scene_max`。
- 支持 `conflict_min_frames`。
- 支持 `conflict_scope`，例如 `target_to_neighbors`、`all_valid_pairs`。
- 在 summary 中单独记录每个子标签，不只记录最终并集标签。

### 7.2 `models/hivt.py`

训练接入处需要做三类调整：

- 将新的 label policy 参数传入 `build_reliability_targets`。
- 训练日志中继续输出 `{mode/fde/miss/conflict/offroad/scene}_risk_target_rate`。
- 在标签修复阶段允许 `rerank_alpha=0` 或只记录 `reranked_pi`，避免塌缩风险分数影响训练判断。

### 7.3 `train.py` 与 `run_single_gpu.sh`

建议增加 CLI 参数或环境变量：

```text
--mode_target_policy
--scene_target_policy
--risk_fde_threshold
--risk_miss_threshold
--risk_conflict_threshold
--risk_conflict_min_frames
--risk_scene_rate_threshold
```

这些参数用于快速跑标签体检和小预算复验，不需要反复改代码。

### 7.4 新增标签体检脚本

建议新增脚本：

```text
scripts/audit_reliability_targets.py
```

脚本功能：

- 加载 train 或 val dataloader。
- 跑若干 batch forward。
- 不反向传播。
- 统计不同阈值和策略下的正样本率。
- 输出表格，帮助选择 `tau_fail`、`tau_col`、`rho`。

至少输出：

```text
fde_risk_target_rate
miss_risk_target_rate
conflict_risk_target_rate
offroad_risk_target_rate
mode_risk_target_rate
scene_risk_target_rate
mode_risk_pred_mean
scene_risk_pred_mean
```

## 8. 损失与训练策略

### 8.1 标签修复前

在标签分布恢复前，关闭复杂训练项：

```text
rank_loss_weight = 0
calib_loss_weight = 0
rerank_alpha = 0 或只做日志评估
```

只保留最基本的 mode risk 和 scene risk 监督。

### 8.2 标签修复后

当标签分布正常、risk head 不再输出全 1 后，再逐步打开：

```text
reliability_loss_weight = 0.5 ~ 1.0
scene_loss_weight = 0.1 ~ 0.3
rank_loss_weight = 0.05 ~ 0.1
calib_loss_weight = 0.0 起步
rerank_alpha = 0.2 -> 0.5
```

`scene_loss_weight` 不宜过大，因为 scene label 比 mode label 更粗，权重过大可能干扰 mode-level 学习。

### 8.3 指标解释顺序

修复阶段不要先看 `minADE` / `minFDE`，应按以下顺序判断：

1. 标签正样本率是否恢复正常。
2. `mode_risk_pred_mean` / `scene_risk_pred_mean` 是否不再接近 `1.0`。
3. `AUROC` / `AUPRC` 是否有基本区分度。
4. `BrierScore` / `ECE` 是否可解释。
5. `reranked_pi` 是否改善 `MR` / `minFDE`。
6. 插件是否没有明显破坏宿主预测性能。

## 9. 实验闭环设计

### 9.1 Exp A：标签体检

不训练，或只跑少量 forward batch，统计：

```text
fde_risk_target_rate
miss_risk_target_rate
conflict_risk_target_rate
offroad_risk_target_rate
mode_risk_target_rate
scene_risk_target_rate
```

目标：

- 主 mode 标签不再全正。
- scene 标签不再全正。
- conflict 标签明显稀疏化。
- off-road 作为弱代理单独观察。

这一步应先于任何正式训练。

### 9.2 Exp B：小预算复验

沿用当前阶段性实验设置：

```text
max_epochs = 1
limit_train_batches = 64
limit_val_batches = 16
batch_size = 8
```

重点检查：

- `val_mode_risk_target_rate`
- `val_scene_risk_target_rate`
- `val_mode_risk_pred_mean`
- `val_scene_risk_pred_mean`
- `val_mode_AUROC`
- `val_mode_AUPRC`
- `val_mode_BrierScore`
- `val_mode_ECE`

该阶段不以 `minADE` / `minFDE` 提升作为主要成功标准，而是确认风险头是否从塌缩中恢复。

### 9.3 Exp C：Reranking 单独验证

使用同一个 checkpoint 分别评估：

```text
original pi
reranked_pi
```

关注：

- `minFDE`
- `MR`
- `minADE`
- top-1 FDE
- failure detection AUROC

如果 reranking 后 `MR` 或 `minFDE` 改善，同时 `minADE` 不明显恶化，可以支持 risk-aware reranking 的价值。

### 9.4 Exp D：Shift Evaluation

在 risk head 有判别性之后，再打开：

- `HistoryDropout`
- `NeighborDropout`
- `StateNoise`
- `MapJitter`

重点验证：

```text
shift 强度上升 -> failure rate 上升 -> risk score 同步上升
```

同时比较 risk score 与原始 `pi` 在失败检测上的差异。

## 10. 验收标准

第一阶段修复完成的最低标准：

- `mode_risk_target_rate` 不再等于 `1.0`。
- `scene_risk_target_rate` 不再等于 `1.0`。
- `conflict_risk_target_rate` 不再等于 `1.0`。
- `mode_risk_pred_mean` 不再接近 `1.0`。
- `AUROC` / `AUPRC` 能正常反映风险区分能力。
- 训练 loss 不出现异常震荡或 NaN。

进一步成功标准：

- risk score 对高 FDE / miss mode 有明显更高响应。
- reranking 后 `MR` 或 `minFDE` 有改善。
- shift 条件下 risk score 随失败率上升。
- 插件不显著恶化宿主模型原始预测指标。

## 11. 推荐实施顺序

### 11.1 第一轮：只修监督信号

1. 新增标签体检脚本，先统计不同阈值下的标签正样本率。
2. 将主 `mode_targets` 改成 FDE/MR failure。
3. 将 `conflict` / `offroad` 暂时移出主并集，只保留日志。
4. 将 `scene_targets` 改成 target-centric 或比例聚合。
5. 跑标签体检，确认 target rate 不再全正。

### 11.2 第二轮：小预算训练闭环

1. 关闭 `rank_loss` / `calib_loss`。
2. 设置较小 `scene_loss_weight`。
3. 跑 `max_epochs=1` 的小预算 plugin 训练。
4. 对照旧实验结果，判断 risk head 是否恢复。

### 11.3 第三轮：验证 risk-aware reranking

1. 使用同一个 checkpoint 对比 `pi` 和 `reranked_pi`。
2. 若 reranking 有改善，再打开小权重 `rank_loss`。
3. 若 reranking 无改善，先检查 risk score 与 FDE 的相关性，不急于扩大训练。

### 11.4 第四轮：恢复论文级实验

1. 加入 shift evaluation。
2. 重新跑 baseline/plugin 对照。
3. 做必要消融。
4. 最后再考虑多 backbone 泛化。

## 12. 论文表述调整

当前阶段不应表述为：

```text
插件显著提升 HiVT 预测精度
```

更稳妥的表述是：

```text
在基本不破坏宿主预测性能的前提下，引入可判别的 mode-level 与 scene-level 失败风险估计，并通过 risk-aware reranking 和 shift-aware evaluation 体现部署价值。
```

论文创新点应聚焦于：

- 自动风险标签协议。
- mode-level + scene-level 双层可靠性。
- interaction-aware conflict proxy。
- shift-aware reliability evaluation。
- risk-aware reranking。
- plug-in 形式接入宿主轨迹预测模型。

## 13. 当前不建议做的事情

在风险标签塌缩尚未解决前，不建议：

- 直接跑长时间正式训练。
- 直接做 QCNet / MTR / Wayformer 泛化。
- 直接加复杂多 head 并提高损失权重。
- 把当前 `AUROC` / `BrierScore` / `ECE` 当作有效可靠性结论。
- 宣称插件已经提升预测精度。

这些工作应在 risk label 和 risk score 恢复可解释性之后再推进。

## 14. 简短结论

当前 idea 的结构闭环已经打通，问题主要集中在自动风险标签协议。下一步最优先的工作是修正 `mode_targets`、`conflict_targets` 和 `scene_targets`，让 reliability supervision 从全正样本恢复为有物理意义、有正负样本、有评估价值的监督信号。

只有完成这一步，后续的 reranking、shift evaluation 和论文主实验才有可信基础。
