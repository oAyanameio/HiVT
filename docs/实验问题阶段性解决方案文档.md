# 实验问题阶段性解决方案文档

## 1. 文档目的

本文档只记录当前实验主线和接下来要做的实验，不再保留已经解决的问题、历史分歧和旧路线。

当前唯一主线是：

```text
freeze-backbone + mode_risk 训练目标改进 + reranking MR 复验
```

## 2. 当前结论

截至 2026-06-29，当前可以直接作为实验前提使用的结论只有这些：

1. 默认入口使用 `freeze-backbone`。
   - 不再把 joint training 当主入口。
   - 当前重点是验证 risk head 本身能否改善 mode selection。

2. `mode_risk` 相对宿主原始 `pi` 有真实增量。
   - naive `pi` 风险代理约为 `AUROC=0.70`、`AUPRC=0.71`
   - `mode_risk` 已达到约 `AUROC=0.87`、`AUPRC=0.84`

3. `mode_risk` 已具备较强的 FDE 排序能力。
   - 与真实 `FDE` 的 `Spearman = 0.7482`

4. 概率空间 reranking 已经能改善连续误差。
   - 当前 `prob_product` 公式可以改善 `minADE / minFDE`
   - 但还没有转化成 `MR` 收益

5. 当前核心卡点已经收敛。
   - 问题不再是“risk score 没学到”
   - 而是“当前训练目标没有把风险分数稳定转化成更好的 top-1 / MR 决策”

6. `scene` 和 `conflict` 都不是当前主实验入口。
   - `scene head` 仍过置信，`scene_ECE ≈ 0.236`
   - `conflict` 暂时降优先级，不继续扩展大矩阵分析

7. 当前方向本身没有偏离。
   - `freeze-backbone -> mode_risk -> reranking -> MR` 这条主链仍然成立；
   - 当前的问题不是换方向，而是把最后一段任务闭环补齐。

## 4. 当前已实现但尚未验证有效的目标改进

本轮已经接入两类目标侧改动，后续实验直接在此基础上继续：

1. 阈值附近样本加权 BCE

```text
mode_risk_threshold_weight_enabled
mode_risk_threshold_weight_radius
mode_risk_threshold_weight_peak
mode_risk_threshold_weight_base
```

作用：

- 提高 `FDE≈2.0m` 附近样本的监督权重；
- 目标是让风险头对是否跨过 MR 阈值更敏感。

2. `top-k + near-threshold` 过滤的 pairwise rank loss

```text
mode_risk_rank_top_k
mode_risk_rank_near_threshold_only
mode_risk_rank_threshold_radius
```

作用：

- 只在更可能影响 top-1 决策的 mode 对上施加排序约束；
- 减少与当前 reranking 目标无关的远离阈值样本干扰。

3. `rank loss` 监督已从 `ADE` 切到 `FDE`

作用：

- 让排序目标和 `MR` / reranking 目标更一致。

4. 因此当前结论只能是：

## 6. 当前未解决问题

当前只保留一个主问题：

```text
如何让 freeze-backbone 下已经具备较强 FDE 排序能力的 mode_risk，
通过合适的决策规则或更对齐的监督，真正转化成不劣化、最好改善的 top-1 / MR 收益？
```

当前更具体的判断是：

1. 只优化整体 `FDE` 排序还不够。
2. 当前瓶颈很可能已不在“风险头能否学到全局 FDE 排序”，而在“这些 risk 分数如何被用于 top-k 决策”。
3. 需要优先减少 `hit -> miss` 翻转，而不是只让一部分已 miss 样本的 FDE 变小。
4. 当前 `mode_target = 1(FDE_k > 2.0m)` 仍然是绝对分类目标，而 reranking 真正要解决的是：

```text
在目标车 top-k 候选里，谁应该保住 top-1，谁应该被替换？
```

这意味着当前训练目标与最终 reranking 任务之间仍有结构性错位。
5. `rank loss` 已确认在当前严格配置下不是完全失活，但监督覆盖偏稀。
6. 最近几轮分析已经足够说明问题大致在哪里，后续重点不再是继续扩展旁支诊断，而是把监督直接收紧到 top-1 决策相关的 mode 竞争上。
7. 即使把 rank loss 放宽到 `top-k only`，当前 `MR` 仍然没有改善，说明问题不只是 “strict 配置过稀”，而是当前监督定义本身仍然没有直接约束 reranking 的正确翻转方向。
8. 当前 `prob_product` 会对所有候选 mode 对称处理，但 `hit -> miss` 与 `miss -> hit` 的机会结构天然不对称；这可能是当前 reranking 净造 miss 的主要来源。
9. `near-threshold` 加权 BCE 继续把权重压在 `FDE≈2.0m` 的硬阈值边界上，可能在放大边界标签噪声；这是当前强怀疑，但还不是已证实根因。

## 7. 接下来实验顺序

### 7.1 第一优先级：先做零训练成本的 reranking rule 反事实扫描

在继续训练前，先在现有 checkpoint 上直接扫描 reranking 决策规则，不再先开新 run。

```text
目标：
在不改风险头参数的前提下，判断是否存在更保守的翻转规则，
使 reranked_MR <= original_MR
```

建议直接在 `eval_reranking.py` 中加入两类规则参数：

```text
--rerank_margin
只有当 risk(pi_top1) - risk(best_alt) > margin 时才允许替换
```

```text
--rerank_guard
如果 risk(pi_top1) 低于某绝对阈值，则无论如何保留原 top-1
```

当前要先回答的不是“再训一轮会不会更好”，而是：

```text
MR 的主瓶颈，到底是在 reranking rule 太激进，
还是在风险头连决策相关的那对 mode 都分不开？
```

判读规则：

- 如果存在某组 `(margin, guard)` 让 `reranked_MR <= original_MR`：
  - 说明当前主瓶颈更偏向 reranking rule；
  - 下一步优先改决策逻辑，而不是先继续训练风险头。
- 如果再保守的规则也救不回 `MR`：
  - 才说明当前风险头确实分不开决策相关的 mode pair；
  - 这时再进入新的 objective 设计，依据会更干净。

### 7.2 第二优先级：基于规则扫描结果，复盘 hit->miss / miss->hit case

如果 7.1 证明保守规则仍然救不回 `MR`，优先做决策相关 case 复盘：

1. `hit -> miss` / `miss -> hit` 数量变化；
2. `pi_top1` 与 `best_alt` 的 risk gap 分布；
3. 哪些错误翻转其实发生在“原 top-1 已经是 hit”的样本上；
4. `FDE≈2.0m` 的边界样本是否确实更容易出现随机翻转。

### 7.3 第三优先级：若规则扫描仍救不回 MR，再改成更直接的 reranking-oriented objective

如果 7.1 / 7.2 之后仍然确认问题在风险头本身，则当前路线应从：

```text
near-threshold FDE shaping
```

转向更直接的：

```text
miss-oriented target / ranking objective
```

这一步再决定是否：

- 引入更直接的阈值分类目标；
- 改写 pairwise ranking 的正负样本定义；
- 或单独约束 top-1 竞争 mode 的边界。

优先考虑的不是“更大范围的全 mode 排序”，而是更贴近 reranking 场景、且带方向性的监督：

```text
对目标车，只监督 current_top1_mode 与 best_alternative_mode 的竞争关系
```

核心原则是：

- 如果当前 top-1 已经是 hit，优先保护它，不要再施加对称的坏翻转机会；
- 如果当前 top-1 是 miss 且存在更好的替代 mode，再施加“允许翻转”的监督；
- 目标不是让所有 mode 都学会统一的绝对风险刻度，而是先把真正影响 top-1 决策的 mode pair 区分开；
- `near-threshold BCE` 不再默认继续加大，应先关闭或把 `peak` 降回接近 `1.x`，避免继续往硬阈值噪声区压权重。

## 8. 暂不展开的方向

以下内容不再作为当前主实验展开项：

1. joint training 回炉
2. scene 作为主卖点扩写
3. conflict 的大规模矩阵展开
4. off-road 重新并入主监督

当前处理原则：

- `scene` 只做输入聚合 / 信息瓶颈检查；
- `scene_loss_weight` 可以在 mode-only 对照里进一步下调，避免噪声干扰；
- `conflict` 只保留低优先级辅助分析；
- 主时间预算集中在 `mode_risk -> reranking -> MR` 这条链上。
